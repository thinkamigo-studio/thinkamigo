WordPress migration guide

From WordPress to open source HTML — a reusable methodology

This guide documents the four-stage methodology for migrating a WordPress site to an open source HTML, CSS and JavaScript framework. It was developed during the thinkamigo.com migration from the dreamtargets GitHub development environment.

The methodology is reusable. For each new migration project, generate a new snag list .md file from the template at the foot of this guide and work through the stages in order. Do not skip the content audit — it is the most critical step and the one most likely to cause post-migration problems.

STAGE 1 Page content — amends and additions Fix the pages before you move them
Critical: Do not migrate broken or incomplete pages. Every page should be content-complete and correctly structured before migration begins.

For each page, check and complete:

  • OG metadata block — full header including canonical URL, og:title, og:description, og:image, Twitter card tags
  • OG image — 1200×630px, page-specific variant or branded default fallback
  • Page title — consistent format: "Page name | Site name"
  • Breadcrumb — present and correct
  • All links — no `#` placeholders, no broken hrefs
  • All images — captions, alt text, correct file paths
  • All CTAs — PDF downloads, external links, play buttons functional
  • Content — up to date, aligned with current brand and proposition

Global checks across all pages:

  • Create one branded OG default image for pages without a specific variant
  • Sitemap.xml — create or update for new site structure
  • robots.txt — create or update
  • 404 page — create a branded error page
Note: Canonical URLs in OG metadata should point to the destination domain (e.g. thinkamigo.com), not the development environment. Update these before migration, not after.
STAGE 2 Content audit Classify every piece of WordPress content before touching anything
Critical: Crawl the live WordPress site with Screaming Frog before doing anything else. This gives you a complete map of every URL, status code, redirect chain, page title and meta description. These are real assets. Losing indexed URLs without 301 redirects will damage search visibility — sometimes permanently.

Step 1 — Crawl with Screaming Frog

  1. Run Screaming Frog against the live WordPress site
  2. Export the full URL list — all status codes, page titles, meta descriptions, redirect chains
  3. Filter to 200 status codes only — these are your live pages
  4. This is your master URL inventory and redirect priority list

Step 2 — Classify every URL

Every piece of WordPress content falls into one of four categories:

Category Description Action
Case study Documented project work. Established rankings. High value. Prioritise in redirect map. Create equivalent page in new framework or confirm existing page covers content. Set up 301 redirect. Do not let return a 404.
Blog post Opinion, commentary, news. Lower Google value. Editorial content. Migrate to Substack, archive, or redirect to nearest equivalent page. Never redirect to homepage.
Substack candidate Thought pieces, essays, methodology content. Belongs on Substack going forward. Republish on Substack first, then set up 301 redirect from old WordPress URL to Substack equivalent.
Archive / retire Outdated, superseded or low-value content. Not indexed or negligible traffic. Let expire gracefully or redirect to nearest relevant page. Do not migrate.

Step 3 — Build the redirect map

  • Document every old WordPress URL and its destination in the project snag list
  • All redirects must be 301 (permanent) not 302 (temporary)
  • Never redirect to the homepage — always use the nearest relevant page
  • Test every redirect after implementation
STAGE 3 Pre-migration technical tasks Prepare the environment before go-live

Path resolution

The Thinkamigo framework uses automatic environment detection in loader.js. The base path is set by hostname — no find and replace needed between development and production environments.

  • Confirm loader.js v23.8 or later is in use — environment detection is automatic
  • No data-base attribute changes needed on body tags
  • No find and replace in header.html or footer.html

Canonical URLs

  • Update all canonical URLs in OG metadata from development domain to destination domain
  • Update all og:url values throughout
  • Do this before uploading — not after

WordPress backup

  • Back up the full WordPress database
  • Back up all WordPress files including uploads folder
  • Store backup off-server before decommissioning
  • Do not delete WordPress until the new site is fully tested and live

Redirect implementation

  • Implement all 301 redirects from the redirect map built in Stage 2
  • Typically implemented in .htaccess (Apache) or nginx config
  • Test all redirects in staging before go-live
STAGE 4 Migration and go-live Sequential, testable, reversible
Critical: Do not decommission WordPress until the new site is fully tested on the live domain. Keep WordPress accessible until you are confident everything is working correctly.

Go-live sequence

  1. Upload all new files to the live server
  2. Verify DNS is pointing correctly to new hosting
  3. Test all pages on the live domain — do not assume development testing is sufficient
  4. Test all authentication flows (e.g. /amigos/ gate and guard) on the live domain
  5. Test all video projector and lightbox functionality
  6. Test all forms and enquiry submissions
  7. Test all 301 redirects from old WordPress URLs
  8. Check all OG metadata renders correctly — use Facebook Sharing Debugger and Twitter Card Validator
  9. Decommission WordPress installation
  10. Submit updated sitemap.xml to Google Search Console
  11. Monitor Google Search Console for crawl errors for at least 30 days post-migration

Post-migration monitoring

  • Check Google Search Console weekly for the first month
  • Watch for 404 errors — these indicate missed redirects
  • Watch for coverage drops — these indicate indexing problems
  • Fix any issues within 48 hours of discovery
